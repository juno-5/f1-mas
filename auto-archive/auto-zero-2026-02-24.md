**[관찰]**

- 3 failures: "no suitable personas found" — infra/NAS queries

  - `8c4916c1` (09:32:38): same query — domain=developers, functions=[]
- **Transient**: same queries succeeded 1 min before and 1 min after
- **Race condition evidence**: at 11:15:18, TWO identical requests — one completed, one failed

**[심층 분석]**
1. 현상 분해: `_select_by_function()` has domain fallback (line 262) that should ALWAYS return personas. Failures were transient, not systematic.
2. 근인 추적: `load()` clears ALL index dicts (line 181-185) BEFORE repopulating. If `entries` is empty (parsing failure from corrupt/partial file during SCP), index becomes empty until next hot reload (30s).
3. 구조적 패턴: Clear-then-populate pattern is vulnerable to empty results. Entries are parsed OUTSIDE the lock, but clear happens INSIDE — empty entries still trigger clearing.
4. 레버리지 포인트: Guard in `load()` prevents clearing when entries is empty; guard in `_load_functions_yaml()` prevents caching empty patterns.

**[가설]**
- Claim: "no suitable personas found" caused by hot reload clearing index during SCP deployment
- Basis: (a) `load()` clears dicts even when entries=[] (b) Same query succeeded before/after (c) Simultaneous requests had different outcomes
- Prediction: Guard prevents all transient persona selection failures
- Null: Failures recur with guard in place
- Risk: low — guard only skips when entries empty AND index already loaded
- Impact: Tier 1 reliability — eliminates 50% of all failures (3/6)

**[실험 설계]**
- 실험: Add empty-guard to `load()` + no-cache-empty to `_load_functions_yaml()`
- 채택: Both queries ("서버 상태 확인해줘", "노드 PC hostname CPU") succeed post-deploy
- 기각: Either query still fails with "no suitable personas found"
- 범위: mas_persona_index.py (2 locations)
- 되돌림: 가능 — guard only adds early return

**[실험 결과]**

**[검증] ✅ Confirmed**

## Cycle #42 — Haiku Synthesis Model Cost Optimization

**[관찰]**
- 53 total requests, 45 completed — synthesis cost avg $0.072/req using sonnet
- Total synthesis spend: ~$2.11 (21.6% of total $9.76)
- Synthesis is simple summarization task — agents already did the deep thinking
- Config already supports `synthesis_model` field (line 813 in `mas_agent_runner.py`)
- Sonnet pricing: $3/$15 per MTok (input/output). Haiku: $0.80/$4.00

**[심층 분석]**
1. 현상 분해: Synthesis takes agent outputs (2-5 agents × ~3-10K chars each) and creates a unified summary. This is a well-structured summarization task with explicit template (Key Agreements, Tensions, Unified Recommendation, Action Items).
2. 근인 추적: Synthesis uses the same model as agents (`claude_model: sonnet`) because `synthesis_model` was set to "sonnet" by default.
3. 구조적 패턴: Over-provisioning pattern — using expensive model for routine task. Synthesis doesn't need creative reasoning or deep analysis; it reorganizes pre-computed outputs.
4. 레버리지 포인트: Single config change (`synthesis_model: haiku`) affects ALL multi-agent requests (30/53 = 56% of requests).

**[가설]**
- Claim: Using haiku for synthesis maintains quality while reducing synthesis cost by >70%
- Basis: (a) Synthesis is structured summarization with explicit template (b) Input content already contains analysis — synthesis just reorganizes (c) Haiku handles summarization well
- Prediction: Haiku synthesis produces comparable quality (structured sections, proper tables, actionable items) at <$0.025/synthesis
- Null: Haiku output is significantly worse (missing sections, wrong conclusions, poor formatting)
- Risk: low — single config revert to restore sonnet
- Impact: Tier 3 efficiency — saves ~$2/day (21% cost reduction on synthesis portion)
